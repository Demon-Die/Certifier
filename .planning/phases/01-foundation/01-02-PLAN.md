  <files>supabase/migrations/002_rls_policies.sql, supabase/migrations/003_realtime.sql</files>
  <action>
Write migration 002_rls_policies.sql with RLS for all tables:

**profiles:**
- SELECT: Users see own profile OR maintainers/admins see all
  - Policy: `auth.uid() = id OR EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role IN ('maintainer','admin'))`
- INSERT: Only via trigger on auth.users (handled in Phase 02)
- UPDATE: Users update own profile; admins update any
- DELETE: Admins only

**contributions:**
- SELECT: Users see own contributions; maintainers/admins see all
- INSERT: Service role only (webhook handler)
- UPDATE/DELETE: Admins only

**badge_claims:**
- SELECT: Users see own claims; maintainers/admins see all
- INSERT: Users insert own 'available' claims (triggered by points threshold); service role for initial creation
- UPDATE: Users update own to 'claimed' with certifier_credential_id; admins any
- DELETE: Admins only

**maintainer_settings:**
- SELECT: Maintainers/admins only
- UPDATE: Admins only

**special_badges:**
- SELECT: All authenticated users
- UPDATE: Admins only (awarding)

**special_nominations:**
- SELECT: Maintainers/admins see all; users see own nominations
- INSERT: Maintainers/admins only (nominate)
- UPDATE: Maintainers/admins only (voting)

Write migration 003_realtime.sql:
- ALTER PUBLICATION supabase_realtime ADD TABLE profiles, contributions, badge_claims;
- (maintainer_settings, special_badges, special_nominations NOT in realtime)
</action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Supabase | Browser uses anon key (public) |
| Server → Supabase | Server uses service_role key (secret) |
| GitHub Webhook → API | HMAC verified, writes via service role |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Spoofing | Anon key exposure | accept | Anon key is public by design, RLS enforces access |
| T-01-06 | Tampering | SQL injection via client | mitigate | RLS policies, parameterized queries only, no raw SQL from client |
| T-01-07 | Info Disclosure | Cross-user data leak | mitigate | RLS policies on ALL tables, test with multiple users |
| T-01-08 | Elevation | Role escalation | mitigate | Role check in RLS uses profiles table (not client-trusted), admin only via service role |
| T-01-09 | DoS | Realtime subscription flood | accept | Supabase handles connection limits |
</threat_model>
